# Product Brief — Article 002

**Experiment (the real objective):** does the organisation behave well when quality is
*uncertain*? Article 001's quality was obvious ("good enough"). 002 deliberately chooses a
topic requiring **judgement**, then runs it through a genuine **review + approval** process
(Reviewer role: Publish / Revise / Reject — never rewrite). We measure *decision quality*,
not just capability.

## Article 002

- **Working title:** Recover Before You Build
- **Audience:** Engineers and tech leads making build-vs-reuse decisions.
- **Core idea:** The cheapest engineering is often the code you already have. Before
  building, ask whether the capability can be *recovered* (already works, just disconnected),
  *completed* (almost works), or genuinely needs to be *built*. Most teams jump to Build.
- **Genuine judgement calls (why this topic):** When is recovery a false economy? When does
  clinging to old code cost more than a rewrite? How do you tell "almost works" from "looks
  like it works"? These have no obvious answer — so the reviewer has real work to do.
- **Constraints:** ~700–900 words, Markdown, first-person engineering voice, no fabricated
  statistics or invented benchmarks. Claims about trade-offs should be reasoned, not asserted.

## Review dimensions (Reviewer answers these)

Is it publishable? Is it accurate? Is the reasoning complete? Are claims supported? Is there
unnecessary complexity? → **Verdict: Publish / Revise / Reject** (+ categorised findings).
