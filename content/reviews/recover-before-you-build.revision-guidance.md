# Revision guidance (Reviewer → Builder handoff)

Derived from the combined blind-review findings. This is the Reviewer's *actionable summary*
for regeneration — NOT a hand-rewrite of the article. The factory revises; the human does not.

1. Write ONLY about recover-vs-build. Do **not** describe any review/approval process or
   Publish/Revise/Reject verdicts (that instruction leaked from the brief last time). [H1]
2. Include at least **two concrete, realistic examples** — e.g. a disconnected-but-working
   module that was reconnected instead of rebuilt; a case where clinging to old code cost
   more than a clean rewrite. [H2]
3. **Answer** the judgement calls explicitly: *when is recovery a false economy?* and *how do
   you tell "almost works" from "looks like it works"?* [H3]
4. 700–900 words; no filler openings ("In the realm of…"), no intro/conclusion repetition. [H4,H5,H7]
